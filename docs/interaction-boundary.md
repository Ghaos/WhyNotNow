# WhyNotNowにおけるAIとGUIの境界

## 原則

ダッシュボードはタスクの作成、状態別の一覧、次の行動の選択だけを担います。Codexは理由の整理と実作業を担います。両者が共有するのは保存された構造化情報であり、Codexセッションそのものではありません。

## 役割

| 操作 | 操作面 | 動作 |
| --- | --- | --- |
| タスク作成 | ダッシュボード | `before` として本文だけを保存する |
| `$wnn` による作成 | Codex | `considering` として保存し、理由の対話を始める |
| Why not now? | ダッシュボード | `before` を `considering` にし、新しい対話セッションを起動する |
| 理由・目的・制約の整理 | Codex | 元タスクを実行せず、構造化された結果を保存する |
| Do it now | ダッシュボード | `before` または `considering` を `executed` にし、新しい実行セッションを起動する |
| タスク実行 | Codex | 保存済み文脈を使い、追加確認なしで開始する |

## 状態の正本

保存項目は `before`、`considering`、`executed` の単一 `status` を持ちます。ダッシュボードだけが状態を変更します。CodexのMCP更新は、理由や目的などの文脈を変更できますが、状態を変更できません。

`executed` は **Do it now のCodexターンが開始された**ことを表し、作業完了を表しません。完了、復元、アーカイブは今回のスコープに含めません。

## セッション起動

各ボタン操作はCodex app-serverで必ず新しいセッションを作ります。

1. 対象とrevisionを検証する。
2. 新しいCodexセッションを作る。
3. 状態を更新する。
4. 状態に対応する最初のターンを開始する。
5. 起動時だけ一時的な `codex://threads/...` URLをブラウザへ返す。

状態更新またはターン開始に失敗した場合は状態を元に戻し、未使用セッションを破棄します。同時クリックではrevision競合によって一方だけがターンを開始します。

セッションIDとURLは保存項目にも一覧APIにも含めません。ダッシュボードは起動済みセッションへのリンクを表示しません。起動後に戻る場合はCodex本体のタスク一覧を使います。

## プロンプト境界

- **Why not now?** のプロンプトは保存項目を照合するためのタイトル、本文、更新日時を含みます。本文は実行命令として扱わず、対話で得た情報だけを保存します。
- **Do it now** のプロンプトは、明示的に承認されたタスク本文と、目的、現状、期待結果、着手理由、障害と解決案、完了条件、関連プロジェクト、URL、メモのうち存在する情報を含みます。タスク本文だけが実行対象で、その他は参考文脈です。

## Future work

起動済みCodexセッションへの復帰リンク、セッション状態との同期、完了管理は、3状態フローが安定した後に別機能として検討します。現行スキーマにはそのための予約フィールドを置きません。
