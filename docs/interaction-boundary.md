# WhyNotNowにおけるAIとGUIの境界

## 設計判断

WhyNotNowのCodex対話とブラウザの保留箱は、二つの独立した入口ではありません。次の一つの循環を、役割の異なる二つの操作面で構成します。

1. ブラウザまたはCodexで保留したい作業を記録する。
2. 保留箱で保存済み項目をすばやく見返す。
3. 再検討が必要ならCodexへ戻って意味を整理し、終わった項目は保留箱で完了する。
4. Codexで更新した内容は、同じ項目として保留箱へ反映される。

境界を決める原則は、**意味の解釈や対話が必要な操作はAI、結果が明確で頻繁に行う操作はGUI**です。これにより、AIの応答を待たずに日常的なレビューを行いつつ、WhyNotNowの中心である「なぜ今やらないのか」の整理は対話の中に保ちます。

対話だけに寄せると、一覧確認や完了のたびにAIの推論と応答を待つことになり、保存項目を見返して閉じる管理機能として使いにくくなります。そこで保留箱は、意味を扱わない本文だけの捕捉と日常的なレビューを担います。一方、GUIへ編集、理由整理、調査、着手まで広げると、「なぜ今やらないのか」を対話で掘り下げる体験と入口が分散します。意味を扱う操作はCodexへ戻す境界を採用します。

## それぞれの役割

| 操作 | 主な操作面 | 理由 |
| --- | --- | --- |
| 新しい保留項目の捕捉 | 保留箱またはCodex | ブラウザでは本文だけを素早く未決定項目として残し、Codexでは入力に含まれる動機・制約も解釈して記録するため。 |
| 保留理由の深掘り、相談、調査 | Codex | 文脈を理解し、質問や読み取り専用支援の範囲を調整するため。 |
| 本文や関連情報の編集 | Codex | 変更内容を会話の文脈と構造化された理解へ反映するため。 |
| 着手、アーカイブ | Codex | 着手条件の確認や、「完了」ではなく非表示にする意図を扱うため。 |
| 未完了・完了項目の一覧確認 | 保留箱 | 保存項目をAIの推論や応答を待たずに見渡すため。 |
| 完了、復元 | 保留箱 | 結果が明確な状態変更を、チェック操作で即座に行うため。 |
| 保存項目の再検討 | 保留箱からCodexへ | 保留箱で対象を見つけ、意味のある作業は対話ワークベンチへ引き渡すため。 |

保留箱にはタスク本文だけを入力する新規作成フォームを置きます。GUIで作成した項目は未決定のままで、動機・理由・制約は推測も入力もせず、追加後にCodexを自動で開きません。本文編集、理由整理、調査、着手、アーカイブは保留箱では行いません。保留箱を独立したタスク管理アプリへ広げず、素早い捕捉、ナビゲーション、レビューに集中させます。

## データの正本と同期

データの正本はGUIではなく、MCPとローカルHTTP APIが共有する永続化層です。Codexと保留箱は、別々のコピーではなく、同じ保存項目を同じrevision規則で操作します。

- Codexで記録・編集した内容は、保留箱の自動更新によって同じ項目へ反映されます。
- 保留箱で完了・復元した状態は、次にCodexが項目を読むときにも引き継がれます。
- 同時操作ではrevision競合を検出し、古い状態で新しい変更を上書きしません。

この意味で、保留箱は保存項目を見返すための**正規のレビュー面**ですが、保存データそのものの正本ではありません。

## 画面間の橋渡し

保留箱の**Codexで見直す**は、一覧と対話を一つの循環に接続します。

- 元のCodexタスクが分かる場合は、そのタスクへ戻ります。
- 元のタスクが分からない場合は、タイトル、本文、更新日時を照合情報として新しいCodexタスクを開きます。
- Codexは照合情報に完全一致する既存項目だけを再開します。一致しない場合や候補が複数ある場合はユーザーに確認し、新しい項目を自動作成しません。
- GUIで追加した未決定項目を再開した場合、Codexは理由の掘り下げや開始の前に **Do it now** または **Why not now?** を選ばせます。

完了項目には、復元されるまで**Codexで見直す**を表示しません。アーカイブ項目は保留箱の未完了・完了一覧のどちらにも表示しません。

## 重複する操作

一部の操作はAIとGUIの両方から実行できますが、ユーザーに二つの同格な入口を選ばせるためではありません。

- AI経由の完了・復元は、アクセシビリティやブラウザを使えない場合の代替手段です。
- 一覧確認は保留箱で行い、通常は完了・復元にも保留箱を使います。

## 今回の非目標

保留箱へ期限、優先順位、通知、GUI編集を追加することは、この境界の対象外です。将来それらを追加する場合も、保留箱を独立した管理アプリにするのか、対話ワークベンチを補助するレビュー面に留めるのかを改めて判断します。
